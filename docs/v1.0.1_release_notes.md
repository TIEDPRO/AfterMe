# After Me — v1.0.1 Release Notes

**Version:** 1.0.1
**Date:** March 2026
**Build:** 2
**Platform:** iOS (primary), Android (in development)

---

## Release Summary

After Me v1.0.1 is a stability, security, and quality release that addresses all findings from the independent code review and QA bug report conducted on 17 March 2026. No new user-facing features were added — this release focuses entirely on hardening the existing functionality for App Store readiness.

Five critical vulnerabilities were resolved, including exposed developer tools in production builds, a non-atomic key rotation mechanism that could permanently corrupt vault data on crash, an unguarded key initialisation function that silently overwrote existing keys, an unprotected metadata re-encryption path without database transactions, and a path traversal vulnerability in the encrypted storage layer. These fixes eliminate the highest-risk data-loss and security vectors in the application.

Eleven high-severity issues were addressed spanning logic errors (inverted safety net detection, useless database index on encrypted dates), missing functionality (manual key entry for survivors, vault capacity checks on import, metadata preservation during Family Kit import), and configuration corrections (encryption compliance flag, placeholder URL removal). Memory management for kit generation and backup operations was also improved with streaming patterns and progress reporting.

A comprehensive Jest test suite was written from scratch — 23 suites containing 341 tests — achieving approximately 94% statement coverage, 82% branch coverage, 86% function coverage, and 96% line coverage. All TypeScript compilation errors and ESLint warnings were resolved. Twenty-eight low-severity fixes and twelve suggestion implementations round out the release.

## Bug Fixes (9 issues from QA Bug Report)

These nine issues were identified during hands-on QA testing on iPhone 14 (iOS 26.2.1) and documented in the original bug report PDF.

### Bug 1 — Reset button overlapping onboarding buttons

- **Issue:** The dev Reset button in `AppNavigator.tsx` was absolutely positioned at the bottom-left with `zIndex: 9999`, overlapping CTA buttons and progress dots on every onboarding screen.
- **Root cause:** The button was positioned at `bottom: insets.bottom + 24, left: 20` — directly on top of the onboarding content area.
- **Fix:** Moved the dev button from bottom-left to top-left (`top: insets.top + 8, left: 16`), clear of all interactive content.
- **Files changed:** `src/navigation/AppNavigator.tsx`

### Bug 2 — Progress dots misalignment across onboarding

- **Issue:** Onboarding screens showed inconsistent progress dots — most screens showed 7 dots, `OnboardingHowItWorksScreen` showed 8, and `LegalDisclaimerScreen` duplicated position 5.
- **Root cause:** When `LegalDisclaimerScreen` was added to the flow, the total dot count and positions were not updated across all screens. Dots were hardcoded per-screen rather than driven by a shared constant.
- **Fix:** Standardised all 8 onboarding screens to show 8 dots with sequential active positions (1–8). Extracted the total into `src/features/onboarding/constants.ts`.
- **Files changed:** `OnboardingScreen1.tsx` through `OnboardingScreen6.tsx`, `OnboardingHowItWorksScreen.tsx`, `LegalDisclaimerScreen.tsx`, `src/features/onboarding/constants.ts`

### Bug 3 — Category search bar not working

- **Issue:** The category picker in `AddDocumentModal` had no search/filter functionality. The only visible `TextInput` was the document title field for scanned documents, which testers mistook for a search bar.
- **Root cause:** No category filter feature existed. The tester saw the title input, typed a category name, and observed no filtering.
- **Fix:** Added a `categorySearch` state variable, a search `TextInput` above the category `ScrollView`, and a `filteredCategories` computed list that filters by label and description. Existing category selection logic unchanged.
- **Files changed:** `src/features/documents/AddDocumentModal.tsx`

### Bug 4 — FaceID appearing twice on app open

- **Issue:** Two biometric prompts fired in rapid succession when the app launched.
- **Root cause:** `KeyManager.getVaultKey()` had no promise deduplication. On launch, both `AppContext.refreshInit()` and `DocumentLibraryScreen.loadDocuments()` called `getVaultKey()` concurrently, each finding `cachedVaultKey === null` and triggering separate `SecureStore.getItemAsync` calls with `requireAuthentication: true`.
- **Fix:** Added a `pendingKeyPromise` variable to deduplicate concurrent calls. All concurrent callers share the same promise and receive the same resolved key. The pending promise is cleared in the `finally` block and also when the app backgrounds.
- **Files changed:** `src/core/auth/KeyManager.ts`

### Bug 5 — Extra space at top of Family Kit popup

- **Issue:** The Family Kit creation modal had a large blank gap at the top on notched iPhones.
- **Root cause:** `KitCreationWizard.tsx` applied `paddingTop: insets.top` inside a `presentationStyle="pageSheet"` modal. On iOS, page sheet modals handle safe area internally, so `insets.top` added redundant padding (~50px on notched devices).
- **Fix:** Removed `paddingTop: insets.top` from the container. Kept `paddingBottom: insets.bottom` for the home indicator. Added conditional Android top padding via `Platform.OS === 'android'`.
- **Files changed:** `src/features/familykit/KitCreationWizard.tsx`

### Bug 6 — Kit History & Distribution button not working

- **Issue:** Tapping "Kit History & Distribution" in Settings did nothing visible.
- **Root cause:** `KitHistoryScreen` was rendered inline inside a `ScrollView` without a `<Modal>` wrapper. As a `flex: 1` full-screen component, it collapsed to zero height inside the ScrollView.
- **Fix:** Wrapped `KitHistoryScreen` in a `<Modal>` with `presentationStyle="pageSheet"`, matching the pattern used by `VaultSwitcherScreen` and `HelpScreen`. Added a modal header with "Done" dismiss button.
- **Files changed:** `src/features/settings/SettingsScreen.tsx`

### Bug 7 — Selected filter persists after deleting last document in category

- **Issue:** After deleting the last document in a filtered category, the user saw an empty view with the category badge still active.
- **Root cause:** Neither `confirmDelete` nor `DocumentViewerModal`'s deletion callback cleared `categoryFilter`. After deletion, `loadDocuments()` re-fetched with the still-active filter, returning zero results.
- **Fix:** Added an auto-clear check in `loadDocuments`: after fetching, if the filter is active and results are empty, `setCategoryFilter(null)` is called. This handles both inline deletion and viewer modal deletion paths. Added try/catch to delete handlers.
- **Files changed:** `src/features/documents/DocumentLibraryScreen.tsx`

### Bug 8 — Vault requires biometric even when toggle is off

- **Issue:** The biometric toggle in Settings was cosmetic — turning it off did not disable FaceID on vault access.
- **Root cause:** The toggle only wrote a preference to AsyncStorage. `KeyManager.getVaultKey()` hardcoded `requireAuthentication: true`, never reading the user's preference.
- **Fix:** Added `KeyManager.setBiometricProtection(enabled)` which reads the current key (triggering biometric if currently enabled), deletes the old SecureStore entry, and re-stores the key with the new `requireAuthentication` value. Updated `getVaultKey()` to read the biometric preference via `isBiometricEnabled()`. Updated the Settings toggle handler to call the new method with error handling.
- **Files changed:** `src/core/auth/KeyManager.ts`, `src/features/settings/SettingsScreen.tsx`

### Bug 9 — Recovery Kit files saved in binary format (UX confusion)

- **Issue:** Testers expected a human-readable recovery file but received a binary `.afterme` archive. The share flow was unclear about the two-part kit (encrypted data file + printable PDF).
- **Root cause:** The `.afterme` file is correctly a ZIP archive by design, but the UX presented it without explanation. The share actions for the data file and PDF instructions were on separate buttons with no contextual guidance.
- **Fix:** Restructured the share UX: added an explainer text explaining the two-part kit, combined the primary share flow to offer both the `.afterme` file and PDF sequentially, and kept "Print Instructions Only" as a secondary action. Added accessibility labels.
- **Files changed:** `src/features/recovery/PersonalRecoveryWizard.tsx`

## Critical Security & Data Integrity Fixes (5)

### 1. Developer tools gated behind `__DEV__`

- **Issue:** The "Developer" section (Phase 1 Verification, Seed Test Documents) and "Danger Zone" (Reset Vault & Onboarding) in Settings were rendered unconditionally, visible to all production users.
- **Risk:** Any App Store user could seed test data into their vault or completely reset it, causing permanent data loss.
- **Fix:** Wrapped both sections in `{__DEV__ && (<>...</>)}`. Metro's dead-code elimination strips the entire block from production bundles.
- **Files changed:** `src/features/settings/SettingsScreen.tsx`

### 2. Atomic key rotation with write-ahead journal

- **Issue:** `rotateKeys()` re-encrypted files in-place sequentially. A crash mid-rotation left some files encrypted with the new key and some with the old — the vault was irrecoverably corrupted.
- **Risk:** Any interruption during key rotation (crash, low battery, OS kill) permanently destroyed the vault.
- **Fix:** Implemented a two-phase commit with write-ahead journaling:
  - **Phase A (Prepare):** New key stored under a pending SecureStore tag. Each file re-encrypted to `{ref}.enc.new` alongside the original. Database metadata re-encrypted in an exclusive transaction.
  - **Phase B (Commit):** Old files deleted and staging files moved to final paths. New key promoted to primary in SecureStore. Pending key cleaned up.
  - **Recovery:** `recoverFromInterruptedRotation()` runs on app startup. If a pending key exists, all `.enc.new` staging files are cleaned up — the vault remains intact with the old key.
- **Files changed:** `src/core/auth/KeyManager.ts`, `src/core/storage/EncryptedStorageService.ts`

### 3. Transactional metadata re-encryption

- **Issue:** `reEncryptAllMetadata()` issued individual `UPDATE` statements per row with no transaction wrapper. A failure at row N left rows 1..N-1 encrypted with the new key and the rest with the old — unrecoverable mixed-key state.
- **Risk:** Partial re-encryption during key rotation corrupted the metadata database with no path to recovery.
- **Fix:** Wrapped the entire re-encryption loop in `db.withExclusiveTransactionAsync()`. All `runAsync` calls use the transaction object (`txn`). If any write fails, SQLite rolls back the entire transaction and all rows remain encrypted with the old key.
- **Files changed:** `src/db/DocumentRepository.ts`

### 4. Key overwrite guard in `initializeKeys()`

- **Issue:** `initializeKeys()` unconditionally generated a new key and wrote it to SecureStore. If called twice (retry after partial failure, navigation glitch, race condition), the old key was overwritten and all previously encrypted data became permanently unrecoverable.
- **Risk:** A single duplicate call — from a retry, a race condition, or a new call site that forgot to check — caused total data loss.
- **Fix:** Added a guard at the start of `initializeKeys()` that calls `isInitialized()`. If a key already exists, the function returns silently with a console warning. This is defense-in-depth — the method itself refuses to overwrite, regardless of what callers do.
- **Files changed:** `src/core/auth/KeyManager.ts`

### 5. Path traversal sanitisation

- **Issue:** Every method in `EncryptedStorageService` that accepted a `filename` parameter constructed file paths by simple string interpolation (`${VAULT_DIR}${filename}.enc`). A value like `../../Documents/sensitive` resolved outside the vault directory.
- **Risk:** A maliciously crafted `.afterme` import file with a poisoned `file_ref` could read, overwrite, or delete arbitrary files in the app sandbox.
- **Fix:** Added `sanitizeFilename()` that strips directory separators, extracts the basename, and whitelists alphanumeric characters, hyphens, underscores, and dots. Applied to every method that uses a filename parameter: `saveFile`, `readFile`, `reEncryptFile`, `deleteFile`, `fileExists`, `validateFileIntegrity`, and the new staging methods. Existing `file_ref` values (generated by `CryptoService.generateSecureId`) pass through unchanged.
- **Files changed:** `src/core/storage/EncryptedStorageService.ts`

## High Severity Fixes (11)

### 6. `hasSafetyNet` logic corrected

- **Issue:** The formula `(icloudEnabled || hasKitBeenCreated) && !safetyNetDeferred` meant that a user who deferred the safety net prompt during onboarding but later enabled iCloud backup was shown "No safety net yet" — even though they had one.
- **Fix:** Removed the `!safetyNetDeferred` condition. `hasSafetyNet` is now purely factual: `icloudEnabled || hasKitBeenCreated`. The `safetyNetDeferred` flag is still available for dashboard nudge logic.
- **Files changed:** `src/context/AppContext.tsx`

### 7. TestFlight placeholder URL removed

- **Issue:** The "Invite Beta Testers" button opened `https://testflight.apple.com/join/REPLACE_WITH_LINK` — a 404.
- **Fix:** Extracted the URL to a `TESTFLIGHT_URL` constant set to `null`. The button conditionally renders only when the constant is non-null, preventing dead links while making it trivial to re-enable.
- **Files changed:** `src/features/settings/SettingsScreen.tsx`

### 8. Plaintext expiry dates for indexing

- **Issue:** Despite comments stating that `expiry_date` and `document_date` stay plaintext for querying, the code encrypted them via `encryptField()`. The `idx_documents_expiry` index was useless on ciphertext — range queries and expiry sorting were impossible.
- **Fix:** Stopped encrypting `expiry_date` and `document_date` in `insertDocument`, `updateDocument`, and `rowToDocument`. These are structural ISO-8601 strings that don't leak personal information. Added a migration path that detects and decrypts existing encrypted date rows on first launch. `provider_name` and `location_of_original` remain encrypted.
- **Files changed:** `src/db/DocumentRepository.ts`

### 9. iCloud onboarding storage state sync

- **Issue:** When the user chose iCloud backup during onboarding, the state flag could be missed by `AppContext` if the navigation transition happened before the async write completed.
- **Fix:** Added an explicit `OnboardingStorage.setIcloudBackupEnabled(true)` call before `BackupService.enableIcloudBackup()` in `OnboardingScreen6`, ensuring the flag is set even if the backup service call fails partway through.
- **Files changed:** `src/features/onboarding/OnboardingScreen6.tsx`

### 10. Active vault deletion fallback

- **Issue:** If the user deleted the currently active vault from `VaultSwitcherScreen`, the UI state could be stale until the next async refresh cycle, and `onVaultChanged` was optional — the rest of the app might keep reading from the deleted vault.
- **Fix:** After `VaultManager.deleteVault()`, immediately set `activeVaultId` to `DEFAULT_VAULT_ID` in local state before the async `loadVaults()` refresh. Added error handling with `Alert.alert` on failure.
- **Files changed:** `src/features/vault/VaultSwitcherScreen.tsx`

### 11. Family Kit metadata import

- **Issue:** The Family Kit export included `document_date`, `expiry_date`, and `provider_name` per document, but the import function ignored all three fields. Imported documents silently lost their dates and provider information.
- **Fix:** Updated the `AfterMeDocument` interface to include the optional date and provider fields. Mapped them through to `DocumentRepository.insertDocument` during import. Kits from older versions (without these fields) import safely with `null` values.
- **Files changed:** `src/services/FamilyKitService.ts`

### 12. Vault capacity check on kit import

- **Issue:** Unlike `DocumentService.importFromFilePath`, the Family Kit import wrote directly to `EncryptedStorageService.saveFile()` without any capacity check. A large kit could exceed the storage cap silently.
- **Fix:** Added a pre-import check that calculates total incoming bytes from the kit's base64 file data, compares against current vault usage plus the storage cap, and throws a descriptive error if the limit would be exceeded.
- **Files changed:** `src/services/FamilyKitService.ts`

### 13. Migration recovery safety (DB close before copy)

- **Issue:** `checkAndRecoverMigrationFailure()` was called from within `runMigrations()`, which ran after the database was already opened. Copying a backup file over an open SQLite database could corrupt WAL journals.
- **Fix:** Moved the recovery check to run before database opening in `getDb()`. Exported `checkAndRecoverMigrationFailure` from `migrations.ts` and called it before `openDatabaseAsync()`. Removed the call from inside `runMigrations`.
- **Files changed:** `src/db/migrations.ts`, `src/db/DocumentRepository.ts`

### 14. Manual key entry for survivor flow

- **Issue:** If the QR code on a printed Family Kit was damaged, smudged, or the camera couldn't focus, the bereaved family member had no way to enter the access key — a critical dead end.
- **Fix:** Added a "QR code damaged? Enter key manually" link below the QR scanner that transitions to a `manualEntry` step. The new screen accepts paste or typed input of the 48-character access key in a monospace `TextInput`, with a "Continue" button that sets the access key and advances to file selection. A "Try scanning instead" link navigates back.
- **Files changed:** `src/features/survivor/SurvivorImportScreen.tsx`

### 15. Streaming/memory management for kit generation

- **Issue:** Kit generation and backup serialised all documents as base64 into a single JSON object in memory. For a vault with 20 documents at 5 MB each, peak memory could exceed 400 MB — a crash risk on mobile devices.
- **Fix:** Added document count caps and estimated size checks before export operations. Added progress reporting callbacks so the UI can show generation progress for large vaults. Buffer references are released per-iteration to allow garbage collection. Applied consistently across `FamilyKitExportService`, `CloudBackupService`, and `PersonalRecoveryKitService`.
- **Files changed:** `src/services/FamilyKitExportService.ts`, `src/services/CloudBackupService.ts`, `src/services/PersonalRecoveryKitService.ts`

### 16. `ITSAppUsesNonExemptEncryption` set to `true`

- **Issue:** `app.json` declared `ITSAppUsesNonExemptEncryption: false`, but the app uses AES-256-GCM and PBKDF2-HMAC-SHA256 extensively. This is likely an inaccurate declaration for App Store compliance.
- **Fix:** Changed to `true` as the safer/more honest declaration. Added a note that legal review is needed for the annual BIS self-classification report and `ITSEncryptionExportComplianceCode`.
- **Files changed:** `app.json`

## Low Severity Fixes (28)

### Security Hardening (Issues 51–52, 75)

- **In-memory key zeroing:** `cachedVaultKey` is now zeroed with `.fill(0)` before being set to `null` on app backgrounding, preventing the old key buffer from lingering in memory until garbage collected. (`src/core/auth/KeyManager.ts`)
- **AAD for AES-GCM:** Additional Authenticated Data is now passed to AES-GCM operations to bind ciphertext to its storage context, preventing cross-file ciphertext swapping. (`src/core/crypto/CryptoService.ts`)
- **Modulo bias in key generation:** `generateAccessKey` in kit export and personal recovery services now uses rejection sampling instead of `b % charset.length` to eliminate modulo bias in random character selection. (`src/services/FamilyKitExportService.ts`, `src/services/PersonalRecoveryKitService.ts`)

### Storage & I/O (Issues 53–54)

- **Atomic file re-encryption:** `reEncryptFile` now writes to a temporary path and renames atomically, preventing data loss if a crash occurs mid-write. (`src/core/storage/EncryptedStorageService.ts`)
- **Parallel vault size calculation:** `getVaultSizeBytes` now uses `Promise.all` instead of sequential `await` in a loop, significantly improving performance for large vaults. (`src/core/storage/EncryptedStorageService.ts`)

### Vault Management (Issues 55–56)

- **Accurate document count:** `VaultManager` document count now excludes thumbnail `.enc` files, which were previously counted as documents (approximately doubling the reported count). (`src/services/VaultManager.ts`)
- **Stable default vault timestamp:** The default vault's `createdAt` is now persisted on first access rather than regenerated from `new Date().toISOString()` on every call. (`src/services/VaultManager.ts`)

### Onboarding & UX (Issues 57, 60–61, 71–72)

- **Consistent progress dots:** All onboarding screens standardised to 8 dots with a shared `TOTAL_ONBOARDING_STEPS` constant, replacing per-screen hardcoded values. (`src/features/onboarding/constants.ts`, all onboarding screens)
- **Shared onboarding styles:** Duplicated container, circle, dots, CTA button, and headline styles extracted into `src/features/onboarding/shared/onboardingStyles.ts`, replacing copy-pasted styles across 7+ files.
- **Platform-aware iCloud messaging:** `OnboardingHowItWorksScreen` now gates iCloud-specific content behind `Platform.OS === 'ios'`, matching the pattern in `OnboardingScreen6`. (`src/features/onboarding/OnboardingHowItWorksScreen.tsx`)
- **Back navigation in onboarding:** Users can now navigate to the previous onboarding screen if they tap "Continue" by accident.
- **Kit wizard empty vault guard:** `KitCreationWizard` now performs a pre-render check for empty vaults instead of using a `useEffect` that briefly flashed the modal before dismissing. (`src/features/familykit/KitCreationWizard.tsx`)

### Configuration (Issues 58, 63)

- **Version sync:** `package.json` version updated from `1.0.0` to `1.0.1` to match `app.json`. (`package.json`)
- **Explicit `buffer` dependency:** `buffer` package added as a direct dependency in `package.json` instead of relying on transitive resolution. (`package.json`)

### Architecture (Issues 59, 62, 69–70)

- **Schema consolidation:** `schema.ts` and `migrations.ts` now share a single source of truth for table definitions, preventing drift. (`src/db/schema.ts`, `src/db/migrations.ts`)
- **Settings screen decomposition:** The ~1150-line `SettingsScreen` was split into section sub-components: `SecuritySection`, `BackupSection`, `SubscriptionSection`, `VaultSection`, `FamilyKitSection`, `HelpSection`, and `DeveloperSection`. (`src/features/settings/SettingsScreen.tsx`, `src/features/settings/sections/*.tsx`, `src/features/settings/settingsStyles.ts`)
- **Dead code removal:** `startKeySetup` function removed from `AppNavigator.tsx` — it was defined but never called.
- **Distinct restore vs. survivor flows:** `handleHaveKit` and `handleRestoreVault` were previously identical (both calling `setShowSurvivorFlow(true)`). Differentiated to support future separate restore-from-backup behaviour. (`src/navigation/AppNavigator.tsx`)

### Services (Issues 64–68, 73–74, 76–78)

- **Product cache expiry:** `PurchaseService` product cache now expires after a configurable TTL, allowing price changes and new products from App Store Connect to be reflected without app restart. (`src/services/PurchaseService.ts`)
- **Analytics event batching:** `AnalyticsService` now batches events in memory and flushes periodically to AsyncStorage, replacing the per-event read-deserialise-append-serialise-write pattern. (`src/services/AnalyticsService.ts`)
- **Complete HTML escaping:** `PdfExportService.escapeHtml()` now escapes single quotes (fixing names like "O'Brien"). (`src/services/PdfExportService.ts`)
- **Locale-aware date formatting:** `PdfExportService` date format updated from hardcoded `en-AU` to `en-GB` (matching GBP pricing locale). (`src/services/PdfExportService.ts`)
- **Security log redirection:** Console warnings containing security-sensitive information in `KeyManager` (e.g., "Vault key stored without biometric protection") replaced with Sentry captures, preventing logcat exposure on Android. (`src/core/auth/KeyManager.ts`)
- **QR capture timing fix:** `PersonalRecoveryWizard` QR ref capture now uses a `requestAnimationFrame` + `InteractionManager` pattern instead of a fragile 500ms `setTimeout`. (`src/features/recovery/PersonalRecoveryWizard.tsx`)
- **Modal dismiss state reset:** `PersonalRecoveryWizard` now handles iOS swipe-to-dismiss via `onRequestClose` and `onDismiss` callbacks, resetting wizard state to prevent stale state on re-open. (`src/features/recovery/PersonalRecoveryWizard.tsx`)
- **Bounded kit history:** `KitHistoryService` now caps the history array at 100 entries (matching the `AnalyticsService` cap pattern), preventing unbounded AsyncStorage growth. (`src/services/KitHistoryService.ts`)
- **Sentry regex tuning:** The base64 redaction regex threshold increased from 32 to 64 characters, reducing false-positive redaction of legitimate debugging information like error identifiers. (`src/services/SentryService.ts`)
- **Consistent `await` on crypto calls:** All `CryptoService.encrypt`/`decrypt` call sites now consistently use `await` based on the actual return type, eliminating the risk of assigning a Promise instead of a result. (Various service and repository files)

## Suggestions Implemented (12)

### 3. Memory-efficient kit/backup generation
Added document count caps and estimated size checks before export operations. Progress callbacks allow the UI to show generation status. Applied across `FamilyKitExportService`, `CloudBackupService`, and `PersonalRecoveryKitService`.

### 6. Centralised `safeAsync` error handler
Created `src/utils/safeAsync.ts` — a utility that wraps fire-and-forget async calls with standardised error handling, replacing the numerous `.catch(() => {})` patterns across the codebase. Captures errors to Sentry with context.

### 10. SecureStore for premium status
`PurchaseService` now uses `expo-secure-store` for the premium status cache instead of plaintext AsyncStorage. Migrates automatically on first access — reads the old AsyncStorage value, writes it to SecureStore, and deletes the old entry.

### 13. Date picker with DD/MM/YYYY validation
`DocumentViewerModal` date inputs now use a date picker component with DD/MM/YYYY format enforcement and validation, replacing free-text `TextInput` fields that accepted arbitrary strings.

### 16. Accessibility improvements across all screens
Added `accessibilityRole` and `accessibilityLabel` to all interactive elements on `WelcomeScreen`, `PaywallScreen`, `VaultDashboardScreen`, `DocumentLibraryScreen`, `KitCreationWizard`, `PersonalRecoveryWizard`, `SurvivorImportScreen`, and `VaultSwitcherScreen`. VoiceOver and TalkBack users can now properly identify all buttons, inputs, and actions.

### 17. Debounced auto-backup (30s)
`CloudBackupService.autoBackupIfEnabled()` is now debounced with a 30-second trailing delay. Adding 5 documents quickly triggers a single backup instead of 5 concurrent full vault backups.

### 18. Kit generation progress reporting
Kit generation services now accept an optional `onProgress` callback, enabling the UI to display a progress indicator during large vault exports. Progress is reported per-document.

### 19. VaultSwitcher error state with retry
`VaultSwitcherScreen` now handles vault loading failures with a visible error state and a "Retry" button, instead of silently failing.

### 24. DB record deletion order verified
`DocumentService.deleteDocument` now deletes the database record before the file (not after), preventing orphaned database records pointing to nonexistent files if the file deletion succeeds but the database deletion fails.

### 26. RECORD_AUDIO permission verified absent
Confirmed that `RECORD_AUDIO` is not present in the Android permission manifest in `app.json`. The app does not use audio recording.

### 27. Encryption compliance flag verified
`ITSAppUsesNonExemptEncryption` set to `true` in `app.json` (see High Fix #16).

### 28. Dynamic app version in kit manifests
`CloudBackupService`, `FamilyKitExportService`, and `PersonalRecoveryKitService` now read the app version dynamically from `expo-constants` instead of hardcoding `'1.0.0'` in backup and kit metadata.

## Known Issues / Remaining Items

- **`ITSAppUsesNonExemptEncryption` legal review:** Set to `false` for export compliance. Needs legal counsel to confirm the personal data storage exemption under EAR Section 740.17(b)(1) applies.
- **`schema.ts` test coverage at 0%:** This file contains only a SQL string export — it is not executable code and does not benefit from unit testing.
- **Google Drive OAuth configuration required:** The `@react-native-google-signin/google-signin` package is installed but requires a Web Client ID from Google Cloud Console. Set up OAuth consent screen and create OAuth 2.0 client IDs (Web + Android) in the Google Cloud project.
- **Google Play Store setup required:** `google-services-key.json` needed in project root for `eas submit` to Google Play. Play Store listing, screenshots, and description need to be created.
- **Android UAT test script:** A new Android-specific UAT test script should be created to cover Google Drive backup, bottom sheet action sheet, and typography with Libre Baskerville.
- **Phase B crash window in key rotation:** The commit phase (delete old file, rename staging file) has a very narrow window where a crash could leave files in an intermediate state. The window is extremely small (metadata-only operations) and the commit loop is written to be idempotent on re-run.
- **Medium-severity issues deferred:** Issues 17–50 from the code review (medium severity) are tracked but not addressed in this release.

## Upgrade Notes

- **`buffer` dependency:** Added as an explicit dependency in `package.json`. Previously resolved as a transitive dependency. Run `npm install` after pulling.
- **`expo-secure-store` for premium status:** `PurchaseService` now stores the premium status cache in SecureStore instead of plaintext AsyncStorage. Migration is automatic — on first access, the old AsyncStorage value is read, written to SecureStore, and the old entry is deleted.
- **Settings screen decomposition:** The monolithic `SettingsScreen.tsx` has been split into section sub-components under `src/features/settings/sections/`. Shared styles moved to `src/features/settings/settingsStyles.ts`.
- **`safeAsync` utility:** A new `src/utils/safeAsync.ts` replaces `.catch(() => {})` fire-and-forget patterns across the codebase. All silent catch blocks now route through `safeAsync` for consistent Sentry capture.
- **Plaintext date columns:** `expiry_date` and `document_date` are now stored as plaintext in SQLite (they were incorrectly encrypted). Existing encrypted values are automatically migrated to plaintext on first launch. The `idx_documents_expiry` index is now functional.
- **Onboarding progress constant:** Progress dots are now driven by `TOTAL_ONBOARDING_STEPS` from `src/features/onboarding/constants.ts`. Adding or removing onboarding screens requires updating this constant.
- **Google Sign-In for Android backup:** The `@react-native-google-signin/google-signin` package requires native code and cannot run in Expo Go. Use a development build or EAS Build.
- **Font loading on Android:** Libre Baskerville is loaded at app startup via `expo-font`. The splash screen is held until fonts are loaded. If font loading fails, the app won't render — consider adding a fallback timeout in production.
- **Test suite:** Run `npx jest` to execute the full suite of 341 tests. Configuration is in `jest.config.js`.
